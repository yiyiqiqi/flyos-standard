# flyOS 编码规范

本文档给出 `flyos-standard` 公开版本中，外部贡献者和试用者最需要理解的编码约定。

## 1. 目标

编码规范的目标不是形式主义，而是：

- 让内核代码更容易阅读；
- 让接口边界更清晰；
- 让后续试用反馈和贡献更容易落地；
- 降低跨文件、跨模块理解成本。

## 2. 基本原则

- 命名直接反映职责。
- 接口与实现分层清楚。
- 注释解释边界、约束和原因，而不是逐句翻译代码。
- 改动优先保持一致性，而不是局部炫技。
- 小步提交、文档同步更新。

## 3. 文件组织

建议保持：

- `.h` 与对应 `.c` 文件按职责放在同一目录；
- 文件名使用清晰的小写下划线风格；
- 一个文件尽量只承担一个主职责。

## 4. 命名约定

- 公共类型使用 `flyos_*_t`
- 公共函数使用 `flyos_*`
- 宏使用 `FLYOS_*`
- 平台和端口相关命名保持可读、可检索

## 5. 注释约定

推荐把注释重点放在：

- 这个对象或函数负责什么；
- 输入和输出约束是什么；
- 哪些边界条件需要特别注意；
- 为什么这样实现，而不是另一种方式。

避免无意义注释，例如把代码本身已经表达清楚的内容再重复一遍。

## 6. 公开分支的额外要求

对于 `flyos-standard` 公开分支：

- 不在公开文档里夸大安全或认证结论；
- 不把内部规划、私有路线或过程稿混入公开说明；
- 行为变化要同步更新对应公开文档；
- 面向试用者的说明优先保证真实、简洁、可执行。
